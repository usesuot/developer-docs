Os SDKs SUOT foram projetados para entregar a melhor experiência de desenvolvimento possível tanto em IDEs modernas quanto com assistentes de IA — nenhum método mágico, tipos explícitos em todo lugar, argumentos nomeados e PHPDoc completo em pt-BR garantem que cada sugestão de autocomplete e cada resposta de um agente de IA seja precisa e contextualizada. *Da ordem ao resultado.*

---

## Por que os SDKs são "AI-ready"

A arquitetura do SUOT SDK não foi pensada apenas para funcionar corretamente — foi pensada para ser **compreendida** por ferramentas de análise estática, IDEs e modelos de linguagem. Cada decisão de design contribui diretamente para a qualidade do autocomplete e das sugestões de IA:

<CardGroup cols={2}>
  <Card title="PHPStan nível máximo" icon="shield-check">
    Todo tipo é explícito e verificado em tempo de análise. Nenhuma IDE ou IA precisa inferir — o shape de cada objeto está declarado com precisão.
  </Card>
  <Card title="Sem métodos mágicos" icon="ban">
    Sem `__get`, `__set` ou `__call`. Toda propriedade e método é declarado explicitamente, permitindo introspecção completa pela IDE e pelo agente de IA.
  </Card>
  <Card title="Argumentos nomeados" icon="tag">
    Toda chamada de método usa named arguments. O Copilot e o Claude sugerem cada argumento com seu nome e tipo correto, eliminando erros de posição.
  </Card>
  <Card title="PHPDoc em pt-BR" icon="book">
    A documentação aparece diretamente no tooltip da IDE em português. Sem precisar abrir uma aba externa para entender o que cada parâmetro significa.
  </Card>
  <Card title="Enums tipados" icon="list">
    Em vez de strings soltas, o SDK usa enums PHP 8.1+. A IA completa os casos do enum em vez de adivinhar um valor de string.
  </Card>
  <Card title="Value Objects" icon="cube">
    `Cnpj`, `Chave`, `Rntrc` e outros Value Objects têm construtores explícitos. A IA sugere os argumentos corretos com validação embutida.
  </Card>
</CardGroup>

## Configurando o PHPStan

O SUOT SDK é desenvolvido e testado com **PHPStan no nível 9 (máximo)**. Para obter o máximo de benefício na sua IDE e evitar que erros de tipo passem despercebidos, configure o PHPStan no mesmo nível no seu projeto:

```yaml
# phpstan.neon
parameters:
  level: 9
  paths:
    - src
    - tests
  checkMissingIterableValueType: true
  checkGenericClassInNonGenericObjectType: true
```

```bash
composer require --dev phpstan/phpstan
vendor/bin/phpstan analyse
```

<Tip>
  Execute o PHPStan como etapa obrigatória no seu pipeline de CI/CD. Como o SDK garante nível 9, qualquer violação de tipo na sua camada de integração será capturada antes do deploy.
</Tip>

## IDEs recomendadas

Nenhum plugin especial é necessário para aproveitar o autocomplete dos SDKs SUOT. Os tipos explícitos, enums e PHPDoc funcionam nativamente em qualquer IDE com suporte a PHP moderno.

<CardGroup cols={2}>
  <Card title="PhpStorm" icon="p">
    IDE de referência para PHP. Lê PHPDoc, enums e types nativamente. O Step Builder do SDK é totalmente navegável com `Ctrl+Click`. Nenhuma configuração adicional necessária.
  </Card>
  <Card title="VS Code + Intelephense" icon="code">
    Instale a extensão **PHP Intelephense** (bmewburn.vscode-intelephense). Suporte completo a PHP 8.3+, enums, named arguments e PHPDoc. Gratuito para uso básico.
  </Card>
  <Card title="Cursor" icon="mouse-pointer">
    IDE baseada em VS Code com IA integrada. Aproveita os tipos explícitos do SDK para gerar código correto inline. Use com Intelephense para máxima precisão de autocomplete.
  </Card>
  <Card title="Zed" icon="zap">
    Editor de alta performance com suporte nativo a LSP para PHP. Configure o PHP Language Server (phpactor ou intelephense) para obter autocomplete e navegação de tipos.
  </Card>
</CardGroup>

<Info>
  Certifique-se de que sua IDE está usando o PHP 8.3+ configurado no seu projeto. Enums, fibers e outros recursos modernos usados pelo SDK requerem que o language server aponte para o binário correto.
</Info>

## Agentes de IA

Os SDKs SUOT foram projetados para que agentes de IA gerem código correto na primeira tentativa — sem necessidade de correções manuais de tipo ou consulta a documentação externa.

### GitHub Copilot

O Copilot funciona diretamente com PHP tipado sem configuração adicional. Com o SDK instalado e indexado no workspace, o Copilot:

- Completa os casos de enum (`ModalidadeFrete::`, `TipoServico::`) corretamente.
- Sugere os named arguments na ordem e com o tipo esperado pelo Step Builder.
- Reconhece Value Objects e sugere `new Cnpj(value: '...')` em vez de uma string solta.

### Claude Code / Cursor

Beneficiam-se especialmente da tipagem do SDK. Quando o agente tem acesso ao código do projeto:

- Usa o client `Cte` (`$cte->issue()`, `$cte->cancel()`, …) e encerra em `handle()`.
- Infere etapas obrigatórias pela cadeia tipada até `handle()`.
- Trata `IssueCteResult` com `isAuthorized()` / `isRejected()` / `failure()` / `retryPolicy()`.

### Windsurf / Outros agentes LSP-aware

Qualquer agente que consuma o Language Server Protocol (LSP) se beneficia automaticamente dos tipos explícitos. Nenhuma configuração adicional é necessária além de ter o SDK instalado via Composer e o PHP LSP ativo.

<Tip>
  Ao pedir a um agente de IA que gere código usando os SDKs SUOT, mencione explicitamente o padrão:

  > "Estou usando o SUOT Fiscal SDK com `$cte->…->handle()`, Value Objects com named arguments, enums tipados, `InfrastructureException` para infra e `Result`/`retryPolicy()` para rejeição fiscal. Gere o código seguindo esse padrão."

  Isso ajuda o modelo a escolher a estrutura correta mesmo sem ter indexado toda a documentação do SDK.
</Tip>

## Exemplo de fluxo com IA

Veja como um ciclo típico de desenvolvimento assistido por IA funciona com o SUOT SDK:

<Steps>
  <Step title="Peça ao agente a emissão">
    Solicite ao seu assistente de IA:

    > "Emita um CT-e rodoviário em homologação com `$cte->issue()…->handle()`, incluindo try/catch de InfrastructureException e tratamento de Result."

    Como a cadeia tipada e o PHPDoc em pt-BR guiam cada etapa, a IA gera um scaffold correto sem inventar classes antigas (`EmitirCTe*`, `CTeBuilder`).

    ```php
    use Suot\Fiscal\Core\Exception\InfrastructureException;
    use Suot\Fiscal\Core\Result\RetryMode;
    use Suot\Fiscal\Core\ValueObject\Uf;

    try {
        $result = $cte->issue()
            ->forOperation('ot-000123')
            ->inHomologation()
            ->numbered(series: '1', number: '12345')
            ->issuedIn(new Uf('RS'))
            ->withOperation(cfop: '5353', nature: 'PRESTACAO DE SERVICO DE TRANSPORTE')
            // … issuedBy / from / to / forShipper / deliverTo / carrying / charging …
            ->handle();
    } catch (InfrastructureException $e) {
        $failure = $e->toFailure('issue');
        // Host: alertar; filar conforme $failure->retryPolicy
        throw $e;
    }
    ```
  </Step>
  <Step title="Trate o Result">
    Peça ao agente para completar a tratativa:

    > "Trate `IssueCteResult` com `isAuthorized()`, `isRejected()`, `failure()` e `retryPolicy()`."

    ```php
    if ($result->isAuthorized()) {
        // Host: persistir accessKey, protocolNumber, submittedDocumentXml
    }

    if ($result->isRejected()) {
        $failure = $result->failure();
        $policy = $result->retryPolicy();
        match ($policy?->mode) {
            RetryMode::AfterCorrection => /* remontar issue() com dados corrigidos */,
            RetryMode::QueryBeforeRetry => /* byAccessKey() antes de reemitir */,
            RetryMode::Never, null => /* definitivo; não reenviar */,
            default => /* filar conforme política */,
        };
    }
    ```
  </Step>
  <Step title="Refine com PHPStan">
    Rode o PHPStan para confirmar que o código gerado está correto:

    ```bash
    vendor/bin/phpstan analyse src/
    ```

    Como o SDK opera no nível 9, qualquer tipo incorreto sugerido pela IA é capturado imediatamente, antes mesmo de executar o código.
  </Step>
</Steps>

## MCP server para consulta de documentação

Se você utiliza agentes compatíveis com o **Model Context Protocol (MCP)**, pode conectar a documentação do SUOT SDK diretamente ao seu agente para que ele responda perguntas sobre a API sem precisar de busca externa.

O [Mintlify MCP](https://mcp.mintlify.com) permite que agentes como Claude e Cursor consultem documentação Mintlify em tempo real via protocolo MCP. Configure o servidor MCP apontando para a documentação do SUOT para que seu agente:

- Encontre o método correto em `$cte` para cada operação sem sair da IDE.
- Consulte exemplos de código diretamente nos tooltips do agente.
- Responda perguntas sobre enums, Value Objects e configuração com base na documentação oficial.

<Info>
  A integração MCP não requer configuração especial nos SDKs SUOT — é uma camada de consulta à documentação, não ao código em si. Consulte a documentação do Mintlify MCP em [mcp.mintlify.com](https://mcp.mintlify.com) para instruções de configuração por agente.
</Info>
